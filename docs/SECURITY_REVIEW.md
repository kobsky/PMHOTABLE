## 2. ROW LEVEL SECURITY (RLS)

### Current RLS Policies

All tables use the same pattern:
```sql
CREATE POLICY "authenticated_users" ON [table]
  FOR ALL
  USING (auth.uid() IS NOT NULL);
```

**What this means:** Any authenticated user can read, write, update, and delete ANY row in ANY table.

| Check | Status | Notes |
|-------|--------|-------|
| RLS enabled | ✅ | Enabled on all 7 tables |
| Policy covers INSERT | ✅ | `FOR ALL` includes insert |
| Policy covers DELETE | ✅ | Any user can delete any row |
| Per-user data isolation | ❌ | No. User A can delete User B's tasks |
| Per-organization isolation | ❌ | No multi-tenant isolation |
| profiles: self-update only | ✅ | Separate update policy: `auth.uid() = id` |

### Risk assessment for a 3-person internal team:

- **Acceptable now:** With 3 trusted team members, the "any authenticated user can delete anything" policy is a UX choice (equal permissions), not a security hole — per CLAUDE.md spec.
- **Risk if scaled:** If the app grows, this RLS strategy is dangerous. A compromised account could delete all data.
- **Recommendation:** Add soft-delete columns (`deleted_at`) so accidental deletes are recoverable. Consider adding an `is_deleted` flag rather than hard deletes.

| SQL injection | ✅ | Supabase SDK uses parameterized queries exclusively |

### Markdown rendering risk

`weekly-view.tsx` renders weekly summaries which are auto-generated by `generate-weekly-summary` edge function. The content is built from DB data (task titles, document names, idea titles). If a user enters `<script>` in a task title, it will be in the markdown which is then rendered. React's JSX rendering should escape this, but if any markdown-to-HTML library uses `dangerouslySetInnerHTML`, this is an XSS vector.

**Action:** Verify the markdown renderer in `weekly-view.tsx` (appears to use a simple split/join approach, not a full markdown parser — confirm and add DOMPurify if using a real parser).

## 4. API / SERVER ACTIONS SECURITY

| Check | Status | Notes |
|-------|--------|-------|
| Auth check on all actions | ✅ | `getAuthenticatedClient()` is called at top of every server action |
| Unauthenticated fallback | 🟡 | Dev mode bypasses auth and returns mock data. If `NODE_ENV=production` is not set correctly, mock data could be served in prod. |
| CORS | N/A | Server Actions use same-origin POST — no CORS needed |
| Edge Function auth | ✅ | Uses `SERVICE_ROLE_KEY` which is server-side only |
| Service role key exposure | ✅ | `SERVICE_ROLE_KEY` is not prefixed with `NEXT_PUBLIC_` so it stays server-side |
| Anon key exposure | ✅ | `NEXT_PUBLIC_SUPABASE_ANON_KEY` is public by design (RLS protects data) |

### Dev mode bypass risk (MEDIUM)

```typescript
// lib/supabase/server.ts
export async function getAuthenticatedClient() {
  if (!process.env.NEXT_PUBLIC_SUPABASE_URL) {
    return null; // triggers mock data in all server actions
  }
  // ...
}
```

If deployed to production with missing env vars, ALL server actions return mock data (empty arrays or fake data) instead of erroring. This is a silent failure mode.

**Fix:** In production, throw an error if Supabase keys are missing instead of returning null.
